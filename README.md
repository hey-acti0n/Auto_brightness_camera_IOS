# DZ — адаптивная яркость видеопотока (iOS)

Мобильное приложение для **iOS** (Xcode): в реальном времени анализируется освещённость сцены с камеры и выполняется **адаптивная коррекция яркости/контраста** кадра. Ядро обработки — **Objective‑C++** (`BrightnessProcessor`), захват — **AVFoundation**, интерфейс — **SwiftUI**.

---

## 1. Общая схема работы программы

1. **Захват** — `AVCaptureSession`, пресет `hd1280x720`, формат пикселей `kCVPixelFormatType_32BGRA`, делегат `AVCaptureVideoDataOutputSampleBufferDelegate` на фоновой очереди.
2. **Два представления кадра**:
   - **Оригинал** — растеризация `CVPixelBuffer` → `CGImage` через `CIContext` → `UIImage` (bitmap), чтобы корректно отображаться в SwiftUI.
   - **Обработанный** — копия BGRA в память, расчёт яркостного канала **Y (BT.709)**, анализ гистограммы, **CLAHE** по Y, **адаптивная γ‑коррекция** по Y, **восстановление RGB** с сохранением цвета, упаковка в `UIImage` через `CGImage` + копия пикселей в `NSData`.
3. **Интерфейс** — полноэкранное превью через `Image(uiImage:)` (чистый SwiftUI, без `UIImageView` в `ZStack`, чтобы слои UI не перекрывались), кнопка «ОРИГИНАЛ / ОБРАБОТАН», индикатор средней яркости в процентах.
4. **Метрики** — после каждого кадра в `FrameMetrics` попадают средняя яркость, доли тёмных/ярких пикселей, контраст, энтропия гистограммы, тип сцены, итоговая γ, время обработки; во SwiftUI — `BrightnessMetrics`.

---

## 2. Математика и алгоритмы

### 2.1. Яркостный канал (luma), стандарт BT.709

Для каждого пикселя BGRA (`B,G,R` — байты 0…2) линейная комбинация:

\[
Y = 0{,}0722\,B + 0{,}7152\,G + 0{,}2126\,R
\]

Результат квантуется в `uint8_t` \[0…255\]. По массиву \(Y\) строится **глобальная гистограмма** \(h[k]\), \(k=0…255\).

### 2.2. Статистики по кадру

- **Средняя яркость**: \(\displaystyle \bar{Y} = \frac{1}{N}\sum_{k=0}^{255} k\,h[k]\), где \(N = W\cdot H\).
- **Среднеквадратичное отклонение**: \(\sigma = \sqrt{\max(0,\,\frac{1}{N}\sum_k k^2 h[k] - \bar{Y}^2)}\). В метриках **контраст** задаётся как \(\sigma/128\) (эвристическая нормировка к \[0…1\]).
- **Доля «тёмных» пикселей**: сумма \(h[k]\) при \(k<50\), делённая на \(N\).
- **Доля «пересвеченных»**: сумма \(h[k]\) при \(k>200\), делённая на \(N\).
- **Нормированная средняя яркость** (для правил и UI): \(\mu = \bar{Y}/255\).

### 2.3. Энтропия Шеннона гистограммы

\[
H = -\sum_{k:\,h[k]>0} p_k \log_2 p_k,\quad p_k = \frac{h[k]}{N}
\]

В коде возвращается **нормированная** величина \(H/8\) (максимум \(\log_2 256 = 8\) бит), то есть диапазон ориентировочно \[0…1\].

### 2.4. Классификация типа освещения

Пороговая логика по \(\mu\), доле ярких пикселей и т.д. (см. `BrightnessProcessor.mm`): тёмная сцена, норма, яркая, переэкспонирование.

### 2.5. CLAHE (Contrast Limited Adaptive Histogram Equalization)

Кадр по яркости \(Y\) разбивается на сетку тайлов размера примерно \(\lceil W/N_g\rceil \times \lceil H/N_g\rceil\) при сетке \(N_g \times N_g\) (параметр `claheTileGrid`).

**В каждом тайле:**

1. Считается локальная гистограмма \(h_i\) по 256 уровням.
2. **Клиппинг**: для каждого уровня \(h_i[k] > L\) избыток переносится в `excess`; \(L\) связан с параметром `claheClipLimit` и числом пикселей тайла:  
   \(L = \max\bigl(1,\,\lfloor c \cdot P / 256\rfloor\bigr)\), где \(c\) — множитель клипа, \(P\) — число пикселей в тайле.
3. Избыток **равномерно** распределяется по всем 256 бинам (с остатком `rem` на первые бина).
4. По гистограмме строится **CDF** и LUT:  
   \(\mathrm{LUT}(v) = \mathrm{round}\bigl(255 \cdot \mathrm{CDF}(v)/P\bigr)\), с ограничением 255.

**На пикселе \((x,y)\):** берутся четыре соседних тайла; для входного уровня \(v=Y(x,y)\) из каждого LUT читаются четыре значения; результат — **билинейная интерполяция** по нормализованным координатам в сетке тайлов (веса \(w_x, w_y\)).

Итог: локальное выравнивание контраста без глобального «выжигания» одного режима яркости.

### 2.6. Адаптивная гамма-коррекция

Сначала **сырая** гамма из целевой средней яркости в нормировке \[0…1\] (как в коде, `meanY` там — это уже \(\mu = \bar{Y}/255\)):

\[
\gamma_{\mathrm{base}} = \frac{\ln t}{\ln y_{\mathrm{safe}}},\quad
t \approx 0{,}48,\quad
y_{\mathrm{safe}} = \mathrm{clamp}(\mu,\,0{,}01,\,0{,}995)
\]

Это решает уравнение \(y_{\mathrm{safe}}^{\gamma} \approx t\) в смысле «подтянуть среднюю яркость к цели» при степенной кривой на \[0,1\].

**Коррекция «перцептроном»** (взвешенная сумма признаков с последующим clamp):

\[
\Delta = w_d d + w_b b + w_c c_{\mathrm{metr}} + w_e e
\]

где \(d,b,c_{\mathrm{metr}},e\) — доля тёмных, доля ярких, контраст, нормированная энтропия; веса фиксированы в коде. Затем \(\Delta\) ограничивается интервалом \([-0{,}15,\,0{,}15]\), итог:

\[
\gamma_{\mathrm{raw}} = \mathrm{clamp}(\gamma_{\mathrm{base}}+\Delta,\,0{,}5,\,1{,}8)
\]

**Временное сглаживание** (анти-мерцание между кадрами), \(\alpha \in [0,1)\) — `temporalSmoothingFactor`:

\[
\gamma_k = \alpha\,\gamma_{k-1} + (1-\alpha)\,\gamma_{\mathrm{raw}}
\]

В коде \(\alpha\) — это как раз `_temporalSmoothingFactor` (чем больше \(\alpha\), тем сильнее «помнит» прошлое).

**Применение к каналу Y:** LUT на 256 уровней:

\[
Y' = \mathrm{round}\bigl(255 \cdot (Y/255)^{\gamma}\bigr)
\]

(реализация через `powf` и таблицу для скорости).

### 2.7. Восстановление цвета (BGRA после изменения Y)

Для каждого пикселя считается «старая» яркость \(Y_{\mathrm{old}}\) по тем же коэффициентам BT.709. Новая яркость после CLAHE+γ — \(Y_{\mathrm{new}}\) из массива `yChannel`.

- Если \(Y_{\mathrm{old}} < 1\) (почти чёрный), R,G,B выставляются в \(Y_{\mathrm{new}}\) (оттенок не определён).
- Иначе **масштабирование с сохранением относительных пропорций RGB**:
  \[
  s = \frac{Y_{\mathrm{new}}}{Y_{\mathrm{old}}},\quad
  R'=\mathrm{clip}(R\cdot s),\; G'=\mathrm{clip}(G\cdot s),\; B'=\mathrm{clip}(B\cdot s)
  \]

Альфа-канал не меняется в этом цикле.

### 2.8. Вывод в UIImage

Пиксели BGRA копируются в `NSData`; создаётся `CGImage` с `kCGBitmapByteOrder32Little | kCGImageAlphaNoneSkipFirst`; `UIImage` получает ориентацию **`.right`**, чтобы соответствовать ландшафтному буферу камеры в портретном UI.

## 4. Этапы разработки и отладки (хронология)

1. **Постановка задачи** — по `Task.md`: адаптация яркости видеопотока в реальном времени с элементами компьютерного зрения (анализ сцены, локальное выравнивание, адаптивная нелинейность).
2. **Ядро CV** — реализация BT.709 → гистограмма → CLAHE с клиппингом и билинейной смесью тайлов → γ по метрикам → восстановление RGB; замер времени кадра.
3. **Интеграция с камерой** — `AVCaptureVideoDataOutput`, очередь обработки, публикация `UIImage` и метрик на главный поток.
4. **Сборка под реальное устройство** — снижение `IPHONEOS_DEPLOYMENT_TARGET` до 18.x, отключение агрессивных флагов Swift 6 / MainActor по умолчанию; переход на `ObservableObject` / `@Published` вместо `@Observable` для совместимости.
5. **Bridging / nullability** — `NS_ASSUME_NONNULL_BEGIN` в заголовке, явный тип `BrightnessProcessor` в `CameraManager`, чтобы Swift не считал инициализатор опциональным.
6. **Ориентация и превью** — согласование буфера камеры (ландшафт) с `UIImageOrientationRight`; отказ от двойного поворота (`videoRotationAngle` + ориентация UIImage).
7. **Чёрный экран после обработки** — исправление: (а) не разделять буфер `CGBitmapContext`/`CGImage` между кадрами; (б) не завышать γ при уже «нормальной» автоэкспозиции телефона; (в) копия пикселей в `NSData` при создании `CGImage`.
8. **Интерфейс и safe area** — многократная итерация: `safeAreaInset`, `GeometryReader`, конфликт с `ignoresSafeArea`; итог — осмысленные отступы по `safeAreaInsets` и упрощение UI по запросу пользователя.
9. **Слои UI под камерой** — причина: `UIViewRepresentable` (`UIImageView`) рисовался поверх SwiftUI; решение: превью на чистом `Image(uiImage:)`.
10. **Чёрный экран в режиме «Оригинал»** — `UIImage(ciImage:)` не растрируется; SwiftUI не рисует такой backing; решение: `CIContext.createCGImage` → bitmap `UIImage`.
11. **Текущее состояние UI** — минимальный оверлей: переключение оригинал/обработка и индикатор средней яркости в процентах.

---

## 5. Сборка и запуск

1. Открыть `DZ.xcodeproj` в Xcode.
2. Выбрать своё устройство (камера не работает в симуляторе без ограничений — лучше физический iPhone).
3. Убедиться, что в настройках таргета указано описание использования камеры (`NSCameraUsageDescription` / ключ в Info).
4. **Product → Run**.
